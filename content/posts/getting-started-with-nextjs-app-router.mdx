When I first heard about the **App Router** in Next.js 13+, I was confused. Wasn't the Pages Router already great? After spending a few weeks with it, I now understand why this is such a big deal — and I want to break it down for you.

## What is the App Router?

The App Router is a new routing system in Next.js that sits inside the `app/` directory. Unlike the old `pages/` directory, it uses **React Server Components (RSC)** by default, which means your components run on the server and send minimal JavaScript to the browser.

The key concepts are:

- **Layouts** — shared UI that wraps pages
- **Pages** — unique UI for a route
- **Server Components** — components that run on the server (default)
- **Client Components** — components that need interactivity (`'use client'`)

## File-Based Routing

Routing is still file-based, but now every route is a folder:

```
app/
├── layout.tsx      ← root layout (wraps everything)
├── page.tsx        ← homepage /
├── about/
│   └── page.tsx   ← /about
└── posts/
    ├── page.tsx   ← /posts
    └── [slug]/
        └── page.tsx ← /posts/:slug
```

## Layouts Are Composable

One of my favourite features is **nested layouts**. Every `layout.tsx` wraps its child routes. The root layout is the shell (HTML, body, navigation), and you can nest layouts inside for specific sections.

```tsx
// app/layout.tsx
export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <Header />
        {children}
        <Footer />
      </body>
    </html>
  )
}
```

## Server vs Client Components

This is the part that took me a while to grasp.

**Server Components** (default) are great for:
- Fetching data directly (no `useEffect` needed!)
- Accessing databases or APIs
- Keeping secrets out of the browser bundle

**Client Components** (`'use client'` at the top) are needed for:
- Event handlers (`onClick`, `onChange`)
- React hooks (`useState`, `useEffect`)
- Browser APIs

A common pattern is to keep the outer shell as a Server Component and extract small interactive bits into Client Components.

## Data Fetching is Just `async/await`

This blew my mind at first. You can just `async/await` inside a Server Component:

```tsx
async function PostPage({ params }: { params: { slug: string } }) {
  const post = await fetchPost(params.slug)  // runs on the server!
  return <article>{post.content}</article>
}
```

No `getServerSideProps`, no `getStaticProps` — just plain async functions.

## What I Learned

After building this website with the App Router, here's what I'd tell my past self:

1. **Default to Server Components** — only add `'use client'` when you actually need interactivity
2. **Layouts are your friend** — use them for shared navigation, fonts, and providers
3. **`generateMetadata`** is the clean way to handle SEO per-page
4. **Static generation is automatic** — pages with no dynamic data are automatically SSG'd

The App Router felt weird at first, but now it feels like the right way to build React apps. Give it a shot!
